# VedaMatch Union Gitabase: дизайн офлайн-читалки

## 1. Цель

Добавить в существующий портал VedaMatch встроенный сервис `/gitabase` для чтения всех 15 книг, представленных на `vedabase.ru` на момент утверждённого импорта. Пользователь входит через текущий аккаунт портала, может читать онлайн, скачивать отдельные книги или всю библиотеку, работать без интернета и синхронизировать прогресс, закладки, заметки и выделения между устройствами.

## 2. Принятые продуктовые решения

- Gitabase является частью текущего `apps/web`, а не отдельным приложением или поддоменом.
- Точка входа сервиса: `/gitabase`.
- Используется существующая авторизация VedaMatch без дополнительного входа.
- Первый релиз включает весь каталог из 15 книг источника.
- Поддерживаются загрузка одной книги и загрузка всей библиотеки.
- Читалка включает оглавление, поиск, прогресс, закладки, темы, размер шрифта, заметки и выделение текста.
- Пользовательские данные синхронизируются через API после восстановления соединения.
- Контент импортируется однократно; автоматическое обновление с источника в MVP отсутствует.
- Контент распространяется в виде версионированных пакетов книг.

## 3. Предпосылка по правам

Реализация исходит из подтверждения владельцем продукта наличия разрешения на импорт, хранение и офлайн-распространение выбранных материалов. Публичная доступность сайта и правила `robots.txt` сами по себе таким разрешением не являются. Для каждого пакета сохраняются URL источника и сведения об атрибуции, чтобы происхождение контента оставалось проверяемым.

## 4. Границы сервиса

Gitabase следует контракту из `docs/service-module-contract.md` и не зависит от модулей Union или будущего Motivation.

### Frontend

- Маршруты: `apps/web/src/app/gitabase/`.
- Компоненты: `apps/web/src/components/gitabase/`.
- API-клиент: `apps/web/src/lib/gitabase-api.ts`.
- Локальное хранилище, загрузчик пакетов, поисковый адаптер и очередь синхронизации находятся внутри пространства Gitabase.
- Общие компоненты портала, авторизация и заголовок переиспользуются без дублирования.

### Backend

- Изолированный модуль: `apps/api/src/modules/gitabase/`.
- Все маршруты имеют префикс `/gitabase` и защищены текущим `AuthGuard`.
- Модуль использует `PrismaService`, текущего пользователя и типы `@vedamatch/shared`.
- Полные тексты книг не записываются в PostgreSQL и не смешиваются с пользовательскими данными.

### Shared types

- Контракты сервиса находятся в `packages/shared/src/gitabase.ts` и экспортируются из `packages/shared/src/index.ts`.
- Типы описывают каталог, манифесты пакетов, локаторы текста, пользовательские сущности и протокол синхронизации.

## 5. Архитектура контента

### Формат пакета

Каждая книга публикуется как каталог версионированных файлов:

- `manifest.json`: идентификатор, название, автор, язык, версия формата, версия контента, размер, контрольная сумма, обложка, атрибуция и список глав;
- `chapters/<chapter>.json`: структурированный текст главы;
- `search-index.json`: сериализованный индекс полнотекстового поиска по книге;
- `assets/*`: изображения, необходимые для офлайн-чтения.

Главный манифест библиотеки содержит 15 книг, размеры загрузок и контрольные суммы. Версия пакета является неизменяемой: исправление создаёт новый идентификатор версии, а не меняет уже опубликованный файл.

### Размещение

Импортёр записывает готовые пакеты в каталог, заданный переменной `GITABASE_CONTENT_DIR`. API публикует манифесты и файлы пакетов через защищённые маршруты с поддержкой потоковой передачи и корректных `Content-Length`, `ETag` и cache-заголовков. Каталог контента разворачивается как отдельный постоянный volume и не добавляется в PostgreSQL или исходный код репозитория.

### Структура текста

Структурная единица чтения содержит стабильный локатор и доступные части исходной страницы:

- оригинальный текст стиха;
- транслитерация;
- пословный перевод;
- литературный перевод;
- комментарий;
- URL канонической страницы источника.

Читалка показывает только существующие части и не создаёт пустые блоки для отсутствующих данных.

## 6. Одноразовый импорт

Импортёр получает список страниц из публичного sitemap и обходит только обычные канонические HTML-страницы. Запрещённые в `robots.txt` маршруты `/ajax.php`, `/next.php`, `/go.php` и `/go_to_search.php` не используются.

Для каждой книги импорт выполняется отдельно:

1. Определить структуру книги, разделов, глав и стихов.
2. Загрузить страницы с ограничением частоты, повторными попытками и паузами при ошибках.
3. Извлечь только основной блок текущего стиха, исключив предварительно загруженный контейнер `#next-contents`.
4. Очистить HTML по строгому allowlist без скриптов, обработчиков событий и внешних embed-элементов.
5. Нормализовать текст и сформировать стабильные локаторы.
6. Построить локальный поисковый индекс.
7. Рассчитать SHA-256 для файлов и пакета.
8. Провести валидацию ссылок, структуры и контрольных сумм.
9. Опубликовать пакет только после успешной проверки всей книги.

Импорт можно безопасно перезапустить: уже полученные страницы кэшируются в рабочем каталоге, а частично собранный пакет не становится доступен пользователям.

## 7. Пользовательский интерфейс

### Библиотека `/gitabase`

- Заголовок и навигация портала.
- Поиск по названиям и метаданным каталога.
- Карточки 15 книг с обложкой, названием, автором, размером и статусом загрузки.
- Действия «Читать», «Скачать», «Продолжить загрузку» и «Удалить с устройства».
- Общая панель «Скачать всю библиотеку» с суммарным размером и прогрессом.
- Фильтры: все, скачанные, не скачанные, начатые и завершённые.

### Читалка `/gitabase/books/[bookSlug]/[chapterSlug]`

- Сворачиваемое оглавление.
- Последовательная навигация между главами и стихами.
- Восстановление последней позиции чтения.
- Настройки светлой, тёмной и сепия-темы, размера шрифта и ширины строки.
- Закладки на стабильный локатор.
- Выделение диапазона текста с цветом.
- Заметка, привязанная к локатору или выделению.
- Поиск внутри текущей книги и по всем скачанным книгам.
- Явный индикатор офлайн-режима и статуса синхронизации.

Полнотекстовый офлайн-поиск работает только по скачанным книгам. Поиск по названиям каталога доступен всегда после первого открытия библиотеки.

## 8. Офлайн-хранение

- Service Worker кэширует оболочку `/gitabase`, необходимые статические ресурсы и страницу восстановления после ошибки.
- IndexedDB хранит манифесты, главы, поисковые индексы, прогресс загрузок и пользовательскую очередь изменений.
- Приложение запрашивает `navigator.storage.persist()` после первого явного скачивания.
- Перед загрузкой проверяется доступная квота, когда браузер предоставляет такую информацию.
- Книга считается доступной офлайн только после проверки всех файлов и контрольных сумм.
- Активация пакета атомарна: незавершённая или повреждённая версия не заменяет рабочую.
- Удаление книги очищает её текст и индекс, но не удаляет синхронизированные заметки, закладки и прогресс.

## 9. Модель пользовательских данных

Модели Prisma используют префикс `Gitabase`:

- `GitabaseReadingProgress`: пользователь, книга, локатор, процент, время последнего чтения и revision;
- `GitabaseBookmark`: стабильный ID, пользователь, книга, локатор, подпись, revision и tombstone;
- `GitabaseAnnotation`: заметка или выделение, диапазон, цвет, текст заметки, revision и tombstone;
- `GitabaseAnnotationRevision`: предыдущие версии текста заметки для восстановления после конфликта;
- `GitabaseSyncMutation`: уникальный `clientMutationId` для идемпотентной обработки повторных отправок.

Связь разрешена только с общей моделью `User`. Связей с моделями Union и других сервисов нет.

## 10. Синхронизация

1. Любое действие сначала применяется к локальной базе и немедленно отражается в интерфейсе.
2. Клиент создаёт мутацию с уникальным ID, типом сущности, ожидаемым revision и полезной нагрузкой.
3. При наличии сети очередь отправляется пакетами в `/gitabase/sync/push`.
4. Сервер идемпотентно применяет новые мутации и возвращает серверные revision.
5. Клиент получает удалённые изменения через `/gitabase/sync/pull?after=<cursor>`.
6. Удаления передаются как tombstone, чтобы они корректно распространялись между устройствами.

Прогресс чтения принимает последнюю подтверждённую сервером позицию. Для закладок и выделений используется последняя серверная revision. При конкурентном изменении текста заметки новая версия становится активной, а предыдущая сохраняется в `GitabaseAnnotationRevision` и доступна для восстановления.

## 11. Ошибки и восстановление

- Потеря сети переводит загрузку в состояние паузы без потери уже проверенных файлов.
- Повторная загрузка продолжает работу с первого отсутствующего или повреждённого файла.
- Недостаточная квота показывает необходимый и доступный объём и предлагает удалить другие книги.
- Ошибка контрольной суммы блокирует активацию пакета и предлагает повторить загрузку.
- Ошибка синхронизации не блокирует чтение и локальное редактирование; очередь повторяется с экспоненциальной задержкой.
- Ошибка одного пакета не влияет на остальные скачанные книги.
- Интерфейс показывает состояния «сохранено локально», «ожидает синхронизации», «синхронизировано» и «ошибка синхронизации».

## 12. Изменение карточки портала

После готовности MVP запись `Service` для `gitabase` меняется:

- `url`: `/gitabase`;
- `status`: `active`.

До прохождения приёмочных проверок карточка остаётся в состоянии `coming_soon`.

## 13. Тестирование

### Unit

- разбор структуры страниц и исключение `#next-contents`;
- очистка HTML и защита от небезопасной разметки;
- построение и чтение манифестов;
- проверка SHA-256;
- операции IndexedDB;
- поиск и стабильность локаторов;
- разрешение sync-конфликтов и идемпотентность мутаций.

### Integration

- API каталога и защищённая выдача пакетов;
- загрузка, пауза, продолжение и удаление книги;
- создание и синхронизация прогресса, закладок, заметок и выделений;
- повторная отправка одной мутации без дублирования данных.

### End-to-end

- вход в портал и открытие `/gitabase`;
- скачивание одной книги и чтение после отключения сети;
- скачивание всей библиотеки;
- создание заметки офлайн и синхронизация после восстановления сети;
- продолжение чтения на втором устройстве;
- восстановление после прерванной загрузки и недостатка места.

Основные целевые платформы MVP: Chrome и Edge на desktop, Chrome на Android. Safari/iOS поддерживается по возможности, но требует отдельной проверки квот и поведения фоновых загрузок.

## 14. Критерии готовности MVP

- Карточка портала открывает `/gitabase` без повторной авторизации.
- В библиотеке отображаются все 15 импортированных книг.
- Пользователь может скачать одну книгу или всю библиотеку и видеть реальный прогресс.
- Полностью скачанная книга открывается после перезапуска браузера без сети.
- Поиск работает внутри книги и по всем скачанным книгам.
- Прогресс, закладки, заметки и выделения работают офлайн и синхронизируются после появления сети.
- Повторная синхронизация не создаёт дубликаты.
- Неполный или повреждённый пакет не помечается как доступный офлайн.
- Сервис не обращается к `vedabase.ru` из браузера пользователя; источник используется только одноразовым импортёром.

## 15. Вне объёма MVP

- автоматическая или периодическая синхронизация с `vedabase.ru`;
- административный интерфейс импорта;
- мобильные приложения Capacitor и desktop-приложение Tauri;
- аудиокниги, социальные комментарии и совместное редактирование заметок;
- полнотекстовый поиск по нескачанным книгам;
- платный доступ, роли и ограничения отдельных книг.
